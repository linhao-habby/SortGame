# SortGame - 色块排序交付游戏开发文档

## 一、项目概述

### 1.1 产品定义
一款基于色块排序 + 订单交付的休闲益智游戏。玩家在操作区的木槽之间移动色块进行排序，当木槽顶部连续同色块满足当前订单要求时自动交付。游戏为无尽模式，随进度推进难度递增。

### 1.2 目标平台
- iOS App（通过 WebForiOS 壳子打包上架 App Store）
- Web 版（GitHub Pages 部署，Chrome / Edge 可直接访问）
- 最低支持：iOS 18+（WebGPU 要求）/ Chrome 113+

### 1.3 视觉风格
- **正面轻微俯视角**（约 11 度俯角），能看到色块的顶面和正面，产生类 3D 立体感
- 深色木纹背景（带竖向纹理 + 暗角效果）
- 色块为**圆角方块**（大圆角 38%，厚实饱满），类 3D 光影（顶面高光 + 正面渐变 + 边缘暗化）
- 木槽为**正面凹槽**造型（深色背板 + 浅棕边框 + 格子分隔线）
- 色彩鲜明、饱满

### 1.4 商业化
- 第一版免费上架，不做内购和广告
- 后续版本可考虑加入广告变现和道具内购

### 1.5 在线地址
- **GitHub Pages**: https://linhao-habby.github.io/SortGame/
- **仓库**: https://github.com/linhao-habby/SortGame

## 二、核心玩法规则

### 2.1 屏幕布局（手机风格居中）

```
┌─────────────────────────────────┐
│         12450  COMBO x3 ██░░    │  ← 分数 + combo进度条
│            ❤❤❤❤❤❤❤❤❤❤          │  ← HP 血条（居中）
│                                  │
│        [🟥 x3]  [🟦 x4]         │  ← 订单（居中浮动，1~3个）
│                                  │
│  ┃绿┃  ┃蓝┃  ┃红┃              │
│  ┃红┃  ┃红┃  ┃蓝┃              │  ← 木槽 + 色块（正面视角）
│  ┃蓝┃  ┃绿┃  ┃绿┃              │
│                                  │
│           [重来]                  │  ← 底部工具栏
│                              ⚙  │  ← 调试面板按钮（右下角）
└─────────────────────────────────┘
```

### 2.2 操作规则
- 点击一个木槽，选中其**最顶部**的色块
- 再点击另一个木槽，将色块移动过去
- 移动的合法条件：
  - 目标木槽未满（色块数 < 槽容量）
  - 目标木槽为**空槽**，**或**目标木槽最顶部色块颜色与待移动色块颜色**相同**
- 可以一次移动**连续同色块**（顶部连续相同颜色的色块一起移动）
  - 如果目标槽剩余空间 < 连续同色块数量，则**只移动能放下的数量**（最少 1 个）

### 2.3 订单系统（渐进式 1→2→3 个订单）
- 订单数量随难度递增：新手阶段 **1 个**订单，中期 **2 个**，后期 **3 个**
- 每个订单内容：需要 N 个同色块（如"绿 x4"、"蓝 x1"）
- 各订单的颜色**不相同**
- **取走总数公式**：`取走总数 = 普通槽数 S - (订单数 - 1)`
  - 1 个订单：取走 S（守恒）
  - 2 个订单：取走 S-1（每轮棋盘多 1 个色块，压力渐增）
  - 3 个订单：取走 S-2（每轮多 2 个，压力陡增）
- 订单由**智能算法生成**：扫描棋盘实际颜色分布，优先选可达性高的颜色，严格保证每个订单需求 <= 该颜色在棋盘上的实际数量

### 2.4 自动交付机制
- **检测时机**：每次玩家完成一次**合法移动操作**后立刻检测
- **触发条件**：某个木槽**顶部连续同色块数量 >= 某个活跃订单的要求数量**，且颜色匹配
- **优先选最多的槽**：同一订单如果多个槽满足条件，选**顶部连续同色数量最多**的槽
- **整段消除**：取走该槽顶部**全部连续同色色块**，即使超过订单要求数量（鼓励堆叠，增强正反馈）
- **单次移动只触发一次交付**：交付完成后不立刻二次检测，需等下一次移动
- **单个订单交付流程**：
  1. 匹配的色块从木槽顶部飞出（整段全部取走）
  2. 色块飞向上方消失（弧线动画）
  3. 该订单标记为 completed（HUD 显示 ✓）
  4. 计算得分（见 2.7 分数系统）

### 2.5 补充机制（两种模式可切换）

**模式 A：每单补充（默认，`REFILL_PER_ORDER: true`）**
- 每完成 **1 个订单**就立即执行一次补充（每个槽底部推入 1 个随机色块）
- 所有订单都完成后 → 生成新一轮订单（不再额外补充）

**模式 B：全部完成后补充（`REFILL_PER_ORDER: false`）**
- 所有订单都完成后才一次性补充 + 生成新订单

**补充规则（两种模式共用）**：
1. **所有木槽**底部各推入 **1 个**随机颜色的新色块
2. 已有色块向上顶 1 格
3. 如果木槽已满，顶部色块**溢出消失**，每溢出 1 个色块**扣 1 点血量**
4. 新色块颜色从当前难度的可用颜色池中随机选取

### 2.6 血量系统
- 初始血量：**10 点**（可通过调试面板调整）
- **扣血条件**：色块补充时，木槽已满导致顶部色块溢出消失，每溢出 1 个色块扣 1 血
- 血量**不可恢复**
- 血量归零 → 游戏结束

### 2.7 分数系统

**得分公式**：
```
单次得分 = (基础分 + 超额分) × combo 倍率

基础分 = 订单要求数量 × BASE_PER_BLOCK（默认 10）
超额分 = 超出数量 × OVERFLOW_PER_BLOCK（默认 15）
combo 倍率 = 1 + (combo - 1) × COMBO_STEP（默认 0.5）
```

**combo 窗口机制**：
- 每次交付 → combo +1，移动计数器归零
- 每次移动未交付 → 移动计数器 +1
- 移动计数器达到 `COMBO_WINDOW`（默认 3）→ combo 重置为 0
- HUD 显示倒退进度条，提示 combo 窗口剩余

**示例**：
| 操作 | 订单 | 实际消 | combo | 倍率 | 得分 |
|------|------|--------|-------|------|------|
| 交付 | 红x3 | 5 | 1 | x1.0 | 60 |
| 移动无交付 | - | - | 进度条减1/3 | - | - |
| 交付 | 蓝x4 | 4 | 2 | x1.5 | 60 |
| 移动无交付 x3 | - | - | combo断 | - | - |

**分数参数均可通过调试面板实时调整。**

### 2.8 游戏结束条件
- **血量归零**：溢出扣血累计达到初始血量
- **死局**：所有木槽已满，且没有任何合法移动可执行

### 2.9 游戏模式
- **无尽模式**：不分关卡，单个连续游戏
- 随着完成订单数量增加，难度递增（在现有棋盘上**动态调整**，不重新生成）：
  - 颜色种类增多（新颜色通过底部推入引入）
  - 新增木槽
  - 订单数量增加（1→2→3）
  - 订单要求的色块数量变化
  - 可用空槽减少
- 玩家的目标：尽可能获得高分

### 2.10 难度递增规则

| 完成订单数 | 颜色种类 | 木槽数 | 容量 | 空槽 | 初始块数 | 订单数 | 每单需求 | 阶段 |
|-----------|---------|--------|------|------|---------|--------|---------|------|
| 0 - 5     | 2       | 3      | 5    | 1    | 3       | 1      | 3       | 新手引导 |
| 6 - 15    | 4       | 4      | 5    | 1    | 4       | 1      | 3 - 4   | 入门 |
| 16 - 30   | 5 - 6   | 5 - 6  | 5    | 1-2  | 4       | 2      | 3 - 4   | 中等 |
| 31 - 50   | 7 - 8   | 6 - 7  | 5-6  | 1-2  | 5       | 2      | 4 - 5   | 进阶 |
| 51 - 80   | 8 - 10  | 7 - 8  | 6    | 1    | 5       | 3      | 4 - 5   | 困难 |
| 80+       | 10 - 12 | 8 - 9  | 6-7  | 1    | 6       | 3      | 5 - 6   | 极限 |

> **所有难度参数均可通过调试面板（右下角齿轮按钮）实时调整、保存到本地、导出为代码。**

### 2.11 辅助功能
- **重新开始**：完全重置（棋盘、进度、血量、分数全部恢复初始值）
- **调试面板**：右下角齿轮按钮，可编辑所有难度参数、分数参数、补充模式
  - 应用并重来：立即生效
  - 保存到本地：写入 localStorage，刷新后自动加载
  - 导出代码：生成可粘贴到 GameConfig.js 的代码
  - 恢复默认：清除本地缓存，恢复代码默认值
- **无撤销操作**
- **无道具/救济机制**（v1 版本）

### 3.2 架构设计

**玩法层与表现层解耦**：
- `game/GameFlowController.js`：玩法流程主控（移动→交付→补充→订单→难度），通过回调通知外部
- `core/Game.js`：模块组装 + 回调接线，将玩法事件连接到动画/UI
- 两个开发者可以独立工作，互不冲突

**回调事件**：
| 事件 | 说明 |
|------|------|

### 3.3 渲染方案

#### 3.3.1 相机设置
- **正面轻微俯视**：正交投影，相机 eye=(0, 4.2, 20) target=(0, 0, 0)，约 11 度俯角
- 能看到色块顶面（亮面）和正面（主色面），产生类 3D 效果
- 相机固定不动

#### 3.3.2 色块渲染
- **圆角方块几何体**：12 条边圆角化（半径 38%，7 段细分），厚度 0.65
- **类 3D 光影着色器**：顶面高光 + 正面渐变 + 边缘暗化 + 选中发光
- **GPU Instancing** 批量渲染

#### 3.3.3 木槽渲染
- **正面凹槽造型**：深色背板 + 浅棕色外框 + 格子分隔线
- 选中：框变亮 + 微浮

#### 3.3.4 背景
- 深棕色木纹（竖向纹理 + 暗角 vignette）

#### 3.3.5 木槽布局
- 木槽在 XY 平面**多列排列**，自动计算行列数
- 布局居中，间距均匀
- 新增木槽时动态调整布局

#### 3.3.6 触摸检测（hitTest）
- 正交投影下 3D→2D 投影精确计算每个木槽的屏幕矩形
- 触摸时 2D 矩形命中检测

#### 3.3.7 动画系统

动画不依赖物理引擎，用 **Tween 插值**实现。动画播放期间锁定用户输入。

| 动画 | 说明 | 时长 |
|------|------|------|
| 色块弹出 | 从木槽口弹出到飞行高度 | 150ms |
| 色块飞行 | 水平弧线飞到目标槽上方 | 300ms |
| 色块落入 | 落入目标槽位（easeOutBack 弹性） | 200ms |
| 木槽选中 | 轻微上浮 + 发光 | 即时 |
| 连续移动 | 多个同色块依次飞出，间隔 80ms | 叠加 |
| 交付飞出 | 色块向中央收拢 + 向上方飞出缩小消失 | 400ms |
| 底部推入 | 新色块从木槽底部推入（所有槽同时播放） | 300ms |
| 顶部溢出 | 溢出色块向上飞出淡出消失 | 250ms |
| 扣血反馈 | 血量 UI 扣除 + 闪红 + 震动 | 300ms |
| 订单完成 | 订单栏金色闪光 + 显示 ✓ | 400ms |
| 游戏结束 | 画面灰度化 + 结算弹窗弹出 | 600ms |
| 浮动得分 | 屏幕中央弹出 "+30 combo x2" 上浮消失 | 1200ms |

## 五、核心数据模型

```
Block {
  color: number        // 颜色编号（0~11）
}
```

### 5.2 Slot（木槽）
```
Slot {
  id: number           // 木槽编号
  blocks: Block[]      // 从底部到顶部的色块数组
  capacity: number     // 当前容量（默认 5，随难度变化）
}

方法：
  topBlock()           // 获取最顶部色块
  topConsecutiveCount() // 获取顶部连续同色块数量
  topConsecutiveColor() // 获取顶部连续同色块的颜色

  isFull()             // 是否已满
  canReceive(color)    // 能否接收指定颜色的色块
  availableSpace()     // 剩余可用空间
```

```
Order {
  color: number        // 需要的颜色编号
  count: number        // 需要的数量
  status: 'active' | 'completed'
}
```

### 5.4 GameState（游戏状态）
```
GameState {
  slots: Slot[]                    // 所有木槽
  orders: Order[]                  // 当前订单（1~3个）
  completedOrders: number          // 已完成订单总数（累计）
  hp: number                       // 当前血量
  score: number                    // 当前总分
  combo: number                    // 当前 combo 连击数
  comboProgress: number            // combo 进度条（0~1）
  selectedSlotIndex: number | null // 当前选中的木槽
  status: 'playing' | 'animating' | 'delivering' | 'gameover'
}
```

### 5.5 ScoreManager（分数管理）
```
ScoreManager {
  score: number              // 总分
  combo: number              // 当前 combo
  maxCombo: number           // 历史最大 combo
  movesSinceDeliver: number  // 上次交付后的移动次数

  onDeliver(orderCount, deliveredCount)  // 交付时计分
  onNoDeliver()                          // 无交付时更新 combo 窗口
  getComboProgress()                     // combo 进度条比例（0~1）
}
```

## 六、订单生成算法

### 6.1 智能订单生成（支持 1~3 个订单）

> 订单在当前棋盘状态上生成，直接扫描棋盘实际颜色分布（不依赖 colorCount 参数）。

1. 确定取走总数 `T = 普通槽数 S - (orderNum - 1)`
2. 扫描棋盘上所有实际存在的颜色及其数量、可达性
3. 选择 `orderNum` 种不同颜色（带权重随机，优先可达性高的颜色）
4. 分配数量：每个订单至少 1，总和 = T，每个 count <= 该颜色在棋盘上的实际数量
5. 安全校验：最终确保所有订单 count <= 实际棋盘色块数
6. 避免与上一轮订单颜色重复（权重降低）

### 6.2 守恒与压力

| 订单数 | 取走 | 补充 | 差值 | 效果 |
|--------|------|------|------|------|
| 1 | S | S | 0 | 守恒，平稳 |
| 2 | S-1 | S | +1/轮 | 色块逐渐堆积 |
| 3 | S-2 | S | +2/轮 | 快速堆积，高压 |

---

## 七、棋盘管理

### 7.1 初始棋盘生成
1. 根据初始难度确定：颜色种类数 N、木槽数 S、木槽容量 C、初始空槽数 E、初始块数 B
2. 可填充木槽数 = S - E
3. 总色块数 = 可填充木槽数 × B（B 可小于 C，不一定填满）
4. 生成色块池：N 种颜色均匀分配
5. 打乱色块池，依次填入可填充木槽
6. 保存初始状态快照（用于"重新开始"）

### 7.2 难度动态调整（在现有棋盘上执行）
当完成订单数达到难度阈值时：
- **新增颜色**：新颜色通过底部推入的色块自然引入
- **新增木槽**：在棋盘中动态添加新的空木槽
- **容量变化**：更新所有木槽的容量上限
- 不清空/不重新生成棋盘
